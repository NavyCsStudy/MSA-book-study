## 1. 느슨하게 결합된 조직

- 느슨하게 결합된 조직은 msa 전환시 높은 결과를 제공할 가능성이 높음.

## 2. 콘웨이의 법칙

> 시스템(단순한 정보 시스템보다 넓게 정의된)을 설계하는 조직은 필연적으로 그 조직의 소통 구조와 닮은 설계를 한다.

- 콘웨이의 법칙은 느슨하게 결합된 조직이 느슨하게 결합된 아키텍처를 만든다는 사실을 보여줌.

### 2.1 증거

#### 느슨하거나 강하게 결합된 조직

- 강하게 결합된 조직
  - 확실하게 정렬된 비전, 목표를 설정한 상용 제품 회사로 대표됨.
 
- 느슨하게 결합된 조직
  - 분산된 오픈 소스 커뮤니티로 대표됨.
 
- 책에서 소개한 논문에서는 느슨하게 결합된 조직이 더 모듈화되고 덜 결합된 시스템을 만든다는 사실을 발견.

#### 윈도 비스타

- 마이크로소트는 조직 구조가 제품의 품질과 가장 관련성 높은 사실을 알아냄.

#### 넷플릭스와 아마존

- 아마존
  - 작은 팀이 큰 팀보다 빠르게 업무를 수행한다는 사실을 알고 있었음.
  - two-pizza team: 피자 두 판이 부족할 정도로 팀이 크면 안 됨.
  - 서비스 전체 수명주기를 소유하는 팀을 추구.
 
- 넷플릭스
  - 아마존 사례를 참고.
  - 작고 독립적인 팀을 중심으로 조직하여 자신들의 서비스를 상호 독립적으로 만듬.
  - 시스템 아키텍처에 맞는 조직 구조를 설계 -> 팀의 자리 배치 계획까지 이어져 소통하는 서비를 가진 팀들은 가까이 앉도록 배치.
 
## 3. 팀 규모

- 소개한 논문에서는 9명 이상인 프로젝트에서 생산성이 가장 낮다는 결과를 냄.
- 소수의 인원이 모두 동일한 결과에 집중하고 의견을 조율, 업무를 조정도 더 쉬움.
- 소규모 팀에서 해야할 일이 많아진다면, 사람을 추가하는 것이 항상 도움이 되지는 않음.

## 4. 콘웨이의 법칙 이해

- 여러 부분에서 빠르고 효율적으로 결정하고 행동하길 바라는 것과 느슨하게 결합된 시스템 아키텍처는 느슨하게 결합된 조직이 큰 도움이 됨.

> 느슨하게 결합되고 잘 캡슐화된 아키텍처에 맞는 조직 구조를 만든다면.
> 첫째. 속도와 안정성을 모두 높이는 동시에 번아웃과 배포의 어려움을 감소시키면서 더 나은 전달 성과를 만들 수 있음.
> 둘째. 엔지니어링 조직 규모를 키우면서 생산성을 선형적으로 증가시키는 것보다 더 크게 증가시킬 수 있음.

- 조직이 커질수록 중앙 집중화는 의사결정의 효율성과 행동 속도를 저해.
- 커다란 조직이 기민하게 움직이려면, 의사결정을 더 많은 자율성을 갖고 운영할 수 있는 조직으로 이양해 책임을 효과적으로 분산해야 함.

## 5. 소규모 팀, 대규모 조직

> 지연된 소프트웨어 프로젝트에 인력을 추가하면 더 지연된다.

- sw 개발에서 더 많은 사람의 투입이 더 빠른 문제 해결을 의미하지는 않음.
- 인력으로 문제 해결 속도를 높이기 위해서는 작업이 병렬 수행 가능해야 함. 병렬 수행 가능해도 작업자간에 작업 흐름의 조정이 필요함.
- sw 개발에서 효율적인 대규모 작업을 위해 가장 많은 비용이 드는 것은 조정 활동읠 필요성.
- 서로 다른 팀 간의 조정이 많을수록 진행 속도는 더 느려짐.

> 피자 두 판 팀은 자율적, 다른 팀들과의 상호작용은 제한적.
> 상호작용시에는 세부 사항까지 상세히 문서화되고 인터페이스가 명확히 정의됨.
> 자신들의 시스템 모든 면을 소유하고 책임짐.
> 주요 목표 중 하나는 의사소통에 드는 간접 비용을 낮추는 것. 회의 수, 조율 지점, 계획 수립, 테스트, 릴리스 등.
> 독립적일수록 팀이 더 신속하게 움직일 수 있음.

## 6. 자율성에 관해

- 작은 팀들은 각자 맡은 업무를 수행할 수 있는 자율성을 보장해야 함.
- 다른 팀과 업무를 지속적으로 조정하지 않고도 가능한 한 많은 작업을 수행할 수 있는 도구를 제공해야 함.
- 작은 팀을 구성할 때 작은 팀의 이점을 이해 없이 모방만하면 원하는 결과를 얻을 수 없음에 주의해야 함.

## 7. 강력한 소유권 대 공동 소유권

- 코드 소유권의 두 가지 기본 형태.

### 7.1 강력한 소유권

- 팀이 코드 변경부터 사용 기술, 배포 플랫폼 등을 결정함.
- 팀이 변경에 더 많은 책임을 지면서 더 높은 자율성을 가짐.
- 단, 다른 팀에 영향을 미치는 결정은 여전히 조정이 필요.

#### 강력한 소유권은 어디까지인가?

- 전체 수명주기 소유권.
- 설계부터 운영, 배포, 폐기까지 모든 책임을 가짐.
- 여러 팀으로 구성된 조직에서 ms 를 활용하는 합리적인 모델.

### 7.2 공동 소유권

- 어느 팀이든 ms 를 변경할 수 있음.
- 인력을 필요한 곳에 배치할 수 있음.
- 효과를 얻기 위해서는 ms 간에 일관성이 필요함.  단, 일관성을 높이는 과정에서 선택권이 줄어들고 팀 간의 복잡한 조정이 필요할 수 있음.
- 조정은 조직적 결합으로 이어지고, 결합된 시스템 설계를 초래할 수 있음.

### 7.3 팀 수준 대 조직 수준

- 팀 수준에서는 공동 소유권을 가지고 효율적으로 협업하길 바랄 수 있음.
- 조직 수준에서는 팀의 자율성을 보장하기 위해 강력한 소유권 모델을 가질 수 있음.

### 7.4 균형 모델

- 공동 소유권 지향일수록 일괄된 작업 수행 방식을 유지하는 것이 중요함.
- 조직이 강력한 소유권을 갖게 될수록 더 많은 최적화가 가능.

## 8. 활성화 팀

- 활성화 팀은 스트림 정렬 팀을 지원하기 위해 일하는 팀.

<img width="660" height="365" alt="image" src="https://github.com/user-attachments/assets/5066c69c-2f89-4dfb-8f5c-90422c25fd61" />

- 각 팀마다 다른 프로그래밍 언어를 사용한다면 팀 간의 업무 전환, 채용 등 다양한 부분에서 문제가 될 수 있음.

<img width="761" height="509" alt="image" src="https://github.com/user-attachments/assets/61f1ad21-9c4d-4462-b2b7-8c1cbac8cb3a" />

- 다양한 선택을 통제하고 싶다면 이에 대해 논의할 능력이 있어야 함. 이러한 논의가 제대로 이뤄질 수 있도록 도움을 주는 지원 그룹이 팀 전체에 걸쳐 작동해야 함.

### 8.1 실천 공동체

- 동료 간의 공유와 학습을 촉진하는 교차 그룹.
- 활성화 팀원은 팀의 일원으로 다른 팀을 돕기 위해 많은 시간을 할애.
- 실천 공동체는 학습을 활성화하는 것이 중점.
- 활성화 팀과 실천 공동체 팀은 효과적으로 협업하며 활성화팀이 필요한 것을 잘 이해하는 데 필요한 중요한 통찰을 제공할 수 있음.

### 8.2 플랫폼

- 스트림 정렬 팀에게는 작업을 수행할 수 있는 셀프 서비스 도구, 플랫폼이 필요함.
- ms 에 대한 요구 상태 관리, 로그 집계, ms 간 인증 인가와 같은 공통 기능을 구현하는 플랫폼을 통해 생산성을 향상시키고 업무량을 크게 늘리지 않고도 팀이 더 많은 책임을 지게 할 수 있음.

#### 플랫폼 팀

- 플랫폼을 실행하고 관리할 누군가가 필요함. 이러한 기술 스택은 전문 지식을 증명할 정도로 복잡할 수 있음.
- 플랫폼 팀의 사용자는 개발자들. 이들은 개발자들과 협업해 그들을 돕고, 피드백과 요구 사항을 수용해야 함.
- 활성화 팀과 마찬가지로 내부 컨설팅처럼 운영되어야 함.

#### 포장된 길

- 작업을 수행하는 방법을 명확하게 소통, 그 작업을 쉽게 수행할 수 있는 메커니즘을 제공한다는 개념.
- 플랫폼을 통해 이러한 작업을 지원해줄 수 있음.
- 핵심은 포장된 길의 사용을 강제하지는 않고, 목적지에 도달하는 쉬운 방법을 제공하는 것.
- 일반 사례를 쉽게 만들고, 정당한 경우 예외를 위한 여지를 남겨두는 것을 목표로 함.

## 9. 공유 마이크로서비스

- 강력한 소유권 모델은 이점이 많지만 여전히 팀 간에 공유하는 ms 는 존재할 수 있음.

### 9.1 너무 어려운 분할

- 하나의 ms 를 분리하는 비용이 높거나 조직에서 가치를 제대로 파악하지 못하고 있는 경우.
- 대규모 모놀리식에서 흔히 발생하는 현상.
- 아키텍처에 더 밀접하게 맞추기 위해 팀을 합치는 것도 고려해볼 수 있음.

### 9.2 횡단 변경 사항

- 일부 횡단 변경 사항은 시스템 근간을 뒤흔들 수 있음.
- 횡단 변경 사항을 제거하기 위해 아키텍처를 재구성하는 것보다 다른 팀에 변경을 요청하거나, pr 하는 것이 더 저렴한 예외적인 유형.

### 9.3 전달 병목 현상

- 한 서비스에서 변경해야 하는 백로그가 크서 다른 팀과 소통에서 병목이 발생할 수 있음.

## 10. 내부 오픈 소스

- 공유 코드베이스, ms 에 더 쉽게 변경 사항을 반영하는 방법.
- 일반적인 오픈 소스에서 소수의 사람들이 핵심 기여자인 동시에 코드 관리인의 역할을 맡음.
- 조직 내부에서도 이러한 패턴이 잘 작동할 수 있음.

### 10.1 핵심 커미터의 역할

- 일관성 유지를 위해서는 일반적인 오픈 소스 관리와 동일한 패턴이 필요.
- 신뢰할 수 없는 동료의 커밋을 허용할지, 허용한다면 얼마만큼의 의사소통을 할 지 결정해야 함.

### 10.2 성숙도

- 서비스가 충분히 성숙하기 전에는 어떤 것이 좋은 것인지 정하지 못한 경우가 많음.
- 서비스가 성숙되어 변경이 거의 없다면 새로운 기여가 도움이 되기 시작함.

### 10.3 도구

- 내부 오픈 소스 모델 지원을 위해서는 pr, 이슈 검토, 배포를 쉽게할 수 있는 도구가 필요.

## 11. 플러그 가능한 모듈식 마이크로서비스

- 여러 팀의 변경 사항을 반영하기 위해 중앙의 ms 에 전달되는 pr 은 잠재적인 문제의 신호.

#### 소유권 변경

- 팀 소유가 아닌 ms 에 지속적으로 pr 을 한다면 소유권을 가져오는 것이 합리적일 수 있음.
- 단, 해당 ms 로의 pr 이 다양한 출처를 가지고 있다면 소유권 변경은 어려울 것.

#### 여러 가지 변형 실행

- 한 가지 방법은 각 팀별 변형을 운영하는 것.
- 코드 중복은 공통 기능을 가진 프레임워크를 제공하여 해결.

- 공통 기능의 업데이트를 위해서 락스텝 릴리스가 필요한 것은 문제점.
- 프레임워크가 비대해질 수 있고, 이를 사용하는 팀의 작업에 제약이 발생할 수 있음.
- 항상 다른 모든 방법을 고려하고 올바른 방법인지 고민한 뒤에 실행할 것.

#### 라이브러리를 통한 외부 기여

- 또다른 변형은 팀별 기능이 포함된 라이브러리를 제공하고, 하나의 공유 ms 에 함께 패키징하는 것.

<img width="503" height="267" alt="스크린샷 2026-02-15 오후 4 24 07" src="https://github.com/user-attachments/assets/64f72b4c-bb3b-4b0f-be3c-c9b859e585b9" />

- ms 를 추가로 실행할 필요 없으나, 팀별로 새로운 기능의 배포가 필요할 때 중앙 팀의 배포 일정을 따라야 함.
- 팀별 라이브러이에서 발생한 버그가 중앙 팀의 운영 환경에 영향을 끼칠 수 있음.

### 11.1 변경 사항 검토

- 내부 오픈 소스 방식을 채택할 때는 리뷰가 핵심 개념.

> 고위험 변경만 승인받는 것은 sw 전달 성과와 상관관계가 없다.
> 승인 프로세스가 없거나 동료가 검토하는 방식을 사용한 팀은 더 높은 sw 전달 성과를 달성했다.
> 마지막으로 외부 기관의 승인을 필요로 하는 팀은 낮은 성과를 나타냈다.

- 동료는 어떻게 변경을 잘할 수 있는지 평가하고 신속히 수행할 가능성이 높음.
- 팀 외부의 개인은 프로젝트의 코드베이스를 파악하고 있지 않기 때문에 그러한 평가를 할 가능성이 떨어짐.

#### 동기식 및 비동기식 코드 검토

- 페어 프로그래밍에서는 지속적으로 변경 사항에 대해 대화하고, 문제가 발견되면 즉시 수정됨.
